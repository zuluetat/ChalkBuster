## Critical Pitfalls

Mistakes that cause rewrites or block the deadline.

---

### Pitfall 1: Bracket Progression Logic Bug (Silent Data Corruption)

**What goes wrong:** Pick advancement logic has an off-by-one or incorrect slot-mapping bug. A team picked to win Round 1, Game 3 appears in the wrong Round 2 slot, or winners do not populate downstream rounds. The bracket appears to work for the first few clicks then breaks silently — picks get overwritten or appear in wrong positions.

**Why it happens:** The 64-team bracket has a non-obvious slot structure. Regions (East/South/Midwest/West) each feed into a specific half of the Final Four. The Left/Right side pairing within each region must be correct. When building slot-index math by hand in vanilla JS, a single index formula error corrupts all downstream matchups.

**Consequences:** Full rewrite of the progression/advancement logic mid-development. State saved to Supabase may be corrupted and unrecoverable. Tim fills out part of his bracket then hits a broken matchup with 12 hours left.

**Prevention:**
- Define the full bracket slot schema as a data structure FIRST before writing any UI code. Map every game: `{ id, round, region, slot, topTeamSource, bottomTeamSource }` where sources are either initial seeds or `winner_of_game_X`.
- Write and test the progression logic (given picks, which teams advance) against a known bracket outcome before connecting to the UI.
- Use a flat array of 63 game slots indexed 0-62 with explicit source references, not computed arithmetic.

**Detection (warning signs):**
- Picking a team in Round 1 causes a different Round 2 game to show the wrong matchup.
- The Sweet 16 or Elite 8 games show TBD when earlier rounds are complete.
- Final Four slots do not match the correct regional winners.

**Phase:** Address in Phase 1 (bracket data model and rendering). Validate before building any UI chrome.

### Pitfall 2: Pre-Generated JSON Data Staleness After First Four

**What goes wrong:** The team data and analysis JSON is pre-generated by Claude on March 17. By March 19 bracket lock, the First Four games (March 17-18) will have been played and two 16-seeds will be eliminated. If the JSON still shows First Four matchup participants as open slots, the bracket will show teams that no longer exist in the tournament.

**Why it happens:** The First Four produces the actual 16-seeds that enter the Round of 64. Pre-generating data before these results are known means the slots for four 16-seed positions are provisional. If Claude pre-bakes specific First Four teams into matchup data, those teams may lose.

**Consequences:** Tim's bracket shows a First Four loser as the 16-seed in a Round of 64 matchup. The analysis is nonsensical. The edit interface for tweaking data becomes load-bearing — Tim must update 4 team entries before submitting.

**Prevention:**
- Treat First Four slots as TBD placeholders in the initial JSON. Build the edit interface for these 4 slots first, before Round of 64 data.
- Alternatively, generate two data variants per First Four game and flag them — Tim picks the winner on March 18 night via the edit interface.
- Document clearly which 4 slots need to be updated after First Four results (March 17-18 games).

**Detection (warning signs):**
- The JSON contains team names like "Mississippi State / Mount St. Mary's" as a single entry instead of one resolved team.
- No mechanism exists to update the 16-seed data after First Four results.

**Phase:** Address in Phase 1 (data model). Document the update workflow before Phase 2 (data generation).

### Pitfall 3: Supabase CORS / Anonymous Auth Misconfiguration on GitHub Pages

**What goes wrong:** The app works perfectly on localhost but fails silently on GitHub Pages. Supabase either rejects requests due to missing allowed origins, or the anon key read/write policies are too restrictive to allow unauthenticated saves.

**Why it happens:** Supabase requires the GitHub Pages domain to be in the allowed origins list. New Supabase projects default to restrictive RLS (Row Level Security) policies. Anonymous saves require explicit INSERT and SELECT policies on the anon role. The GitHub Pages URL format (https://username.github.io/repo-name) is easy to misconfigure or forget entirely.

**Consequences:** Tim deploys the app Wednesday morning, tests the save button, bracket does not persist. No time to debug. Supabase console errors are only visible in DevTools — not obvious to the user.

**Prevention:**
- Add the GitHub Pages URL to Supabase allowed origins BEFORE writing the save/load code, so every test confirms it works in the production environment.
- Write the RLS policy as the first Supabase setup step: anon role can INSERT and SELECT on the brackets table. Test with a curl command or Supabase dashboard before building the frontend integration.
- Use Supabase anonymous auth (supabase.auth.signInAnonymously()) to get a stable user ID for the bracket record. This also enables the shareable link feature naturally.

**Detection (warning signs):**
- Works on localhost but 401/403 errors appear in DevTools console on GitHub Pages.
- supabase.from('brackets').insert() returns an error object instead of data.
- Supabase dashboard shows 0 rows in the table after clicking Save.

**Phase:** Address in Phase 3 (Supabase integration). Test against GitHub Pages URL from day one of Supabase work.

### Pitfall 4: Bracket Visualization Layout Breaking at 1280px Desktop Width

**What goes wrong:** The full 64-team bracket is extremely wide. A naive CSS layout either requires horizontal scroll (bad UX), renders teams too small to read, or breaks entirely on 1280px screens. The ESPN aesthetic requirement means the bracket must look polished — cramped or overflowing layouts fail that bar.

**Why it happens:** A 64-team bracket with 6 rounds has approximately 7 columns of content. At 1280px with typical padding, each column gets approximately 150px — barely enough for a team name. Developers underestimate the layout math and discover the problem late when actual team names are populated.

**Consequences:** The bracket visualization — the centerpiece of the product — looks broken or unreadable. Implementing a completely different layout approach (region tabs instead of all-at-once view) at that point is expensive.

**Prevention:**
- The project already calls for region-by-region tabs (East, South, Midwest, West). This is the correct decision — lean into it hard. Each region tab shows only 15-16 games across 4 rounds, which fits comfortably at 1280px.
- Do not attempt to render all 63 games simultaneously on one screen.
- Use the tab view as the primary bracket interaction surface. Add a "full bracket overview" view as a secondary, lower-priority feature.
- Prototype the single-region layout with real team name lengths before styling the full bracket.

**Detection (warning signs):**
- Any CSS layout plan that assigns less than 200px per team matchup card.
- Bracket mockups drawn at 1920px that have not been checked at 1280px.
- Team name truncation with text-overflow: ellipsis appearing in the base layout.

**Phase:** Address in Phase 1 (bracket rendering). The tab-per-region structure must be the foundational layout decision, not an afterthought.

### Pitfall 5: PDF/Image Export Blocking or Breaking Other Features

**What goes wrong:** PDF/image export is added late and either does not work (html2canvas CORS issues with team logos, jsPDF rendering bugs) or requires significant refactoring of the bracket DOM structure to be exportable. It becomes the last feature attempted and the one most likely to be cut or broken.

**Why it happens:** html2canvas requires all images to be same-origin or have CORS headers. Team logo images from external CDNs will not render. jsPDF has layout fidelity issues with complex CSS. The bracket DOM structure optimized for screen interaction is often not the same structure that exports cleanly.

**Consequences:** The export feature either ships broken (logos missing, layout garbled) or blocks the deadline because it is not isolated — it touches the core bracket DOM.

**Prevention:**
- Treat PDF/image export as a nice-to-have that is completely isolated from Phase 1-3. Do not let it block MVP launch.
- If implementing: use window.print() with a print-specific CSS stylesheet instead of html2canvas/jsPDF. Print CSS is more reliable, produces decent PDF output via browser print-to-PDF, and has zero dependencies.
- Pre-compress and inline team logos as base64 data URIs in the JSON data to avoid CORS issues if html2canvas is used.
- Define MVP done as: bracket fills, saves, and shares. Export is bonus.

**Detection (warning signs):**
- Export feature is being worked on before the bracket rendering, save/load, and share features are complete.
- Team logos are loaded from external URLs (not same-origin assets).

**Phase:** Defer to final phase, explicitly time-boxed. If deadline pressure is high, cut it.

### Pitfall 6: Shareable Link Leaks Editable Picks or Breaks on Direct Navigation

**What goes wrong:** The shareable read-only view link either (a) gives the recipient the ability to edit Tim's bracket, (b) returns a blank page when opened directly without Tim's session, or (c) exposes the Supabase record ID in a way that allows enumeration.

**Why it happens:**
- (a) The read-only flag is only enforced in the UI, not in the data layer — anyone with the link who opens DevTools can call supabase.from('brackets').update().
- (b) The app loads bracket data from Supabase only when a session exists, not when a share token is present in the URL — direct navigation produces an empty bracket.
- (c) Using sequential integer IDs directly as share tokens would allow guessing other brackets (UUIDs are fine, just worth noting).

**Consequences:** Friends who click the share link see a blank page. Or they accidentally overwrite Tim's picks.

**Prevention:**
- Use a separate share_token (random UUID or nanoid) as the URL parameter, not the Supabase row ID.
- On page load, check for ?share=TOKEN in the URL. If present, fetch the bracket by share token and render in read-only mode (no click handlers on team cards).
- Enforce read-only at the RLS level: the brackets table allows SELECT by anyone with the share token, but UPDATE only by the owner session.
- Test the share link in an incognito window before launch.

**Detection (warning signs):**
- Share link URL contains the Supabase UUID directly (?id=abc-123 instead of ?share=randomtoken).
- The app does not have a distinct view mode — it is the same component as the edit mode.
- No incognito window test in the QA plan.

**Phase:** Address in Phase 3 (Supabase integration) when designing the data schema. The share token must be in the schema from day one.

### Pitfall 7: Analysis Data Quality — Contradictory Signals

**What goes wrong:** The pre-generated matchup analysis contains confidence tiers and recommendations that conflict with each other. For example, "High confidence: Duke wins" alongside "Upset alert: opponent has >30% win probability." The tool produces noise instead of signal.

**Why it happens:** Pre-generating analysis for all 63 games means the recommendation engine's logic (confidence tiers + upset alerts + contrarian flags) can produce contradictory signals if not carefully defined before generation. There is no feedback loop to catch inconsistencies.

**Consequences:** Tim does not trust the tool. The analysis feels like noise rather than signal. The core value proposition — data-informed picks — is undermined.

**Prevention:**
- Define the signal hierarchy before generating data: Upset Alert overrides Confidence Tier. If a game has an Upset Alert, the confidence tier for the favorite should be Medium at most, not High.
- Add a timestamp/data-as-of field to the JSON so Tim knows when the data was generated.
- For later-round games (Sweet 16+), pre-generate analysis for the most likely matchups based on seedings and label them clearly as "projected matchup."
- Focus analysis density on Round 1 and Round 2 — that is where Tim needs it most and where the data is freshest.

**Detection (warning signs):**
- A game shows both HIGH CONFIDENCE and UPSET ALERT simultaneously with no explanation.
- Round 3+ matchup cards do not clarify that teams are projected, not confirmed.
- No data-as-of timestamp anywhere in the UI.

**Phase:** Address in Phase 2 (data generation). Define conflict resolution rules before generating the full 63-game JSON.

### Pitfall 8: Plain JS State Management Gets Spaghetti Fast

**What goes wrong:** Without a framework, bracket state (all 63 picks, current region tab, which matchup card is expanded, save status) gets managed with global variables and scattered DOM manipulation. After the third feature is added, tracing a bug requires reading the entire script.

**Prevention:**
- Define a single state object at the top of the script on day one: `{ picks: {}, activeRegion: 'East', expandedGame: null, saveStatus: 'idle', shareMode: false }`.
- Write one render() function that derives the full UI from state. Mutate state, then call render. Never mutate the DOM directly without going through state.
- This is the poor-man's React pattern — it works well for this scale and is easy to reason about.

**Phase:** Establish in Phase 1 before writing any UI interaction code.

### Pitfall 9: GitHub Pages Base Path Misconfiguration

**What goes wrong:** Assets (team logos, CSS, JS) load fine when served from the root (/) but 404 on GitHub Pages because the repo is served at /repo-name/. Relative paths work; absolute paths starting with / break.

**Prevention:**
- Use only relative paths for all assets (e.g., ./assets/logo.png, not /assets/logo.png).
- Test the GitHub Pages URL before showing it to friends — not just localhost.

**Phase:** Address in Phase 4 (deployment). Add to pre-launch checklist.

---

### Pitfall 10: ESPN Aesthetic Becomes a Time Sink

**What goes wrong:** Chasing the ESPN visual style consumes 40% of development time on CSS details (gradients, animations, fonts, card shadows) at the expense of functional completeness. The bracket looks great but the save/share feature is not done on Wednesday morning.

**Prevention:**
- Time-box the aesthetic phase. Define ESPN-aesthetic done as: dark header with red accent, team cards with seed badges, region tabs, and confident typography. Stop there.
- Build the functional skeleton first (bracket picks work, data loads, save/load works), then apply the skin.
- Use CSS custom properties for theming so color/font changes do not require touching every component.

**Phase:** Visual polish should come after Phase 3 (all features working). If the deadline is at risk, cut polish, not features.

### Pitfall 11: Supabase Free Tier Pause After Inactivity

**What goes wrong:** New Supabase free tier projects pause after a period of inactivity (the exact threshold has varied — verify current policy). For a one-time event tool used for 2 days, this means the save/load feature could stop working within a week or two of launch.

**Prevention:**
- For a 2-day MVP, this is fine — the tournament deadline is before the pause would kick in.
- After the tournament, the saved bracket URL may stop loading. Set the expectation that the tool is ephemeral.
- Export bracket state to a JSON file as a local backup before Supabase pauses.

**Confidence:** MEDIUM — Supabase free tier pause policy has changed over versions; verify current behavior in Supabase docs during Phase 3.

**Phase:** Note in Phase 3 documentation as a known limitation.

### Pitfall 12: Hover-Only Interactions Break on iPad

**What goes wrong:** Desktop-first means click handlers. iPad compatibility means touch events. If analysis card expand relies on hover as the only trigger, it breaks on iPad (no hover state). The bracket picks may work (click = tap), but secondary interactions like tooltip/card expansion fail on touch devices.

**Prevention:**
- Use click events exclusively — never hover as the only trigger for critical information.
- Hover can show supplemental info on desktop. All critical interactions must be click/tap accessible.
- Test on an actual iPad before declaring iPad compatible.

**Phase:** Address in Phase 1 (interaction model). Do not build hover-only UI patterns.
