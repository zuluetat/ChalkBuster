# Project Research Summary

**Project:** ChalkBuster — March Madness Bracket Analysis Tool
**Domain:** Static sports analysis web app with Supabase persistence
**Researched:** 2026-03-17
**Confidence:** MEDIUM-HIGH

## Executive Summary

ChalkBuster is a single-user bracket analysis tool with a hard Wednesday-morning deadline. Research across stack, features, architecture, and pitfalls points to one consistent theme: the constraints are the strategy. Plain HTML/CSS/JS with pre-generated JSON data and Supabase for persistence is not a compromise — it is the correct technical approach for a 48-hour deadline. No build tooling, no framework, no runtime AI calls. Everything that can be baked in should be baked in. The result is a fast, deployable, zero-dependency static site with one persistence concern.

The recommended approach centers on data-first development. The JSON schema for teams, matchups, and bracket slots must be defined and locked before any UI code is written. All 63-game analysis (stats, confidence tiers, pros/cons, contrarian flags, upset alerts) is pre-generated by Claude and committed as static files. This eliminates runtime latency, API costs, and rate-limit risk during the bracket-lock crunch. The bracket interaction model follows the pick-propagation pattern used by ESPN and CBS: click a team, it advances, re-picking clears downstream picks.

The key risks cluster around two areas. First, bracket progression logic is easy to get silently wrong — a bad slot-index formula corrupts picks downstream without obvious errors. This must be addressed with a fully-specified 63-game slot schema as a data structure before any rendering code exists. Second, Supabase configuration (CORS allowed origins, RLS policies, share token schema) must be set up against the production GitHub Pages URL from day one, not patched after the app is working on localhost. These two risks, if addressed in Phase 1 and Phase 3 respectively, do not threaten the deadline.

## Key Findings

### Recommended Stack

The stack is nearly fully constrained by the project spec. Vanilla JS with ES modules, CSS custom properties, and GitHub Pages hosting are non-negotiable. Within those constraints, the key library decisions are: Supabase JS v2.99.2 via CDN for persistence (anon key, no auth UI, localStorage UUID for Tim session), Chart.js v4.5.1 for win probability bar charts in matchup cards, html-to-image v1.11.13 (not html2canvas — abandoned) plus jsPDF v4.2.1 for optional export. All libraries load via CDN in a defined order before the app entry point.

**Core technologies:**
- Vanilla JS (ES2022 modules): bracket logic and UI — no build step, no framework overhead
- Supabase JS v2.99.2: bracket persistence and shareable links — free tier, anon key is correct pattern
- Chart.js v4.5.1: win probability bar charts in matchup cards only — lightweight, CDN-friendly
- html-to-image v1.11.13: DOM-to-PNG capture for export — active replacement for abandoned html2canvas
- jsPDF v4.2.1: wraps PNG into downloadable PDF — latest stable as of 2026-03-17
- GitHub Pages: static hosting — constraint and correct choice; zero-config deploy on push

**Do not use:** html2canvas (unmaintained since Jan 2022, 970+ open issues), React/Vue/Svelte (build tooling kills the 48h deadline), D3.js (overkill for two-value bar charts).

### Expected Features

The feature landscape is well-understood. Table stakes are the core bracket interaction loop plus the analysis layer that differentiates ChalkBuster from generic bracket fillers. The key MVP dependency chain: team data JSON must be complete before any analysis features can be built or displayed.

**Must have (table stakes):**
- Full 64-team bracket with click-to-pick advancement — this IS the product
- Region tabs (East/South/Midwest/West) — required for usability at 1280px desktop
- Seed records and seeding displayed on each team card
- Win probability per matchup — users expect a who-is-favored signal
- Save/load bracket state via Supabase
- Shareable read-only link
- Team data: KenPom, SOS, record, spread, public pick %

**Should have (competitive differentiators):**
- Pre-generated Claude analysis per matchup (reasoning prose) — no bracket tool offers this free
- Confidence tier labels (High/Medium/Low) — PoolGenius charges for this
- Contrarian opportunity flags (model confidence vs. public pick %) — actionable pool strategy
- Upset alert badges (>30% win probability for lower seed)
- Side-by-side pros/cons cards per team per matchup
- ESPN-style dark sports aesthetic

**Defer to after MVP launch:**
- PDF/image export — use browser print dialog as fallback until functional features are shipped
- Region-by-region strategy narrative text
- Full ESPN aesthetic polish — function before form
- First Four handling: treat 4 slots as editable placeholders if games unresolved at build time

### Architecture Approach

ChalkBuster follows a three-layer pattern: static JSON data layer, a pure-JS state and logic layer, and DOM rendering components. The central design constraint is a single state.js object as the source of truth for all picks. Renderers read state; they do not maintain their own. Components communicate via custom DOM events (matchup:selected, pick:made) to avoid coupling bracket renderer to matchup card. All data is pre-fetched from local JSON at init — no async calls during interaction.

**Major components:**
1. data/*.json (teams, matchups, bracket) — pre-generated static data, read-only at runtime; everything depends on this shape being stable
2. js/state.js — in-memory bracket state, single source of truth; all renderers read from it, none write directly to DOM
3. js/bracket.js — core bracket render and click-to-pick; most complex component, drives the primary UX
4. js/matchup.js — side-by-side analysis card; reads pre-baked matchup data from state, zero async calls
5. js/supabase.js — persistence adapter; debounced auto-save (500ms) and shareable link generation
6. js/export.js — isolated, time-boxed; bracket DOM capture via html-to-image and jsPDF

**Critical path for build order:** data/ schema lock -> state.js -> bracket.js. If JSON schema changes after bracket.js is built, expect rework.

### Critical Pitfalls

1. **Bracket progression logic bug** — Slot-index math errors silently corrupt picks in downstream rounds. Prevention: define the complete 63-game slot schema as static data with explicit winner_of_game_X source references before writing any JS. Test progression logic against a known outcome before connecting to UI.

2. **First Four data staleness** — First Four games play March 17-18. Pre-generated JSON showing eliminated teams in Round of 64 slots makes the analysis nonsensical. Prevention: treat the 4 First Four slots as editable placeholders from day one; build the data-update workflow for those slots before generating the rest.

3. **Supabase CORS / RLS misconfiguration on GitHub Pages** — App works on localhost, fails in production with silent 401/403 errors. Prevention: add the GitHub Pages URL to Supabase allowed origins and write RLS policies (anon INSERT and SELECT) as the very first Supabase setup step, before any frontend integration.

4. **Shareable link exposing editable picks** — Share link gives recipient ability to modify picks, or shows blank page on direct navigation. Prevention: use a separate share_token UUID column (not the row ID) in schema from day one; enforce read-only at RLS level; test in incognito before launch.

5. **Contradictory analysis signals** — Pre-generated JSON shows High Confidence and Upset Alert simultaneously, undermining trust in the tool. Prevention: define signal hierarchy rules (Upset Alert caps confidence at Medium) before generating any matchup JSON.

## Implications for Roadmap

Based on combined research, a 4-phase structure follows directly from the dependency chain and pitfall warnings.

### Phase 1: Data Foundation and Bracket Core

**Rationale:** Everything depends on the JSON schema being stable. Bracket progression logic is the highest-risk component. Both must be locked before any other work begins. The layout decision (region tabs, not full-bracket-at-once) is foundational — changing it later is expensive.
**Delivers:** Locked JSON schema for all 3 data files, 63-game slot structure with correct progression logic, working click-to-pick bracket with region tabs, single-region layout validated at 1280px
**Addresses:** Full bracket visualization, click-to-pick, region tabs, seed display, team slot data
**Avoids:** Pitfall 1 (progression logic bug), Pitfall 2 (First Four staleness), Pitfall 4 (layout breaking at 1280px)

### Phase 2: Data Generation and Analysis Layer

**Rationale:** Once the JSON schema is locked and bracket rendering is proven, generate all 63-game analysis data. This phase is Claude-heavy and data-quality-focused, not UI-heavy. Signal hierarchy rules must be defined before generation begins.
**Delivers:** Complete teams.json (68 teams with KenPom, SOS, record, spread, public pick %), complete matchups.json (all R64 matchups with confidence tiers, pros/cons, upset alerts, contrarian flags, reasoning text), validated data with no contradictory signals
**Addresses:** Win probability, confidence tiers, contrarian flags, upset alerts, pros/cons cards, KenPom/SOS/spread display
**Avoids:** Pitfall 7 (contradictory signals — signal hierarchy defined before generation)
**Uses:** Static JSON pattern from STACK.md; Chart.js for win probability bars

### Phase 3: Persistence and Sharing

**Rationale:** Supabase integration is isolated and comes after bracket interaction is stable. Schema must include share_token from the start — retrofitting it is a rewrite. Must be built and tested against the GitHub Pages URL, not localhost only.
**Delivers:** Working save/load via Supabase, shareable read-only link with ?share=TOKEN param, read-only render mode (no click handlers when in share mode), Supabase RLS policies enforced at data layer
**Addresses:** Save bracket state, shareable read-only link
**Avoids:** Pitfall 3 (CORS misconfiguration), Pitfall 6 (share link leaking editable picks)
**Uses:** Supabase JS v2.99.2, anon key pattern, debounced auto-save (500ms)

### Phase 4: Polish and Optional Export

**Rationale:** Visual polish and export are explicitly deferred until functional features are done. Export is time-boxed and cuttable. Aesthetic work is strictly bounded to avoid the time-sink pitfall.
**Delivers:** ESPN-style dark aesthetic (dark header, red accent, seed badges, confident typography), optional PDF/image export if time permits
**Addresses:** ESPN aesthetic, export to PDF/image (optional)
**Avoids:** Pitfall 5 (export blocking deadline), Pitfall 10 (ESPN aesthetic rat hole)
**Cut if behind:** Export feature; fall back to window.print() with print CSS

### Phase Ordering Rationale

- Phase 1 before Phase 2: JSON schema must be stable before generating 63 games of data — regenerating data after a schema change is expensive
- Phase 1 before Phase 3: Bracket state model must be stable before building the Supabase persistence adapter
- Phase 2 before Phase 4: Data must be complete before matchup cards can be styled
- Phase 3 before Phase 4: Functional features (save/share) must ship before aesthetic polish
- Export deferred to Phase 4 and explicitly time-boxed: the single biggest deadline risk is export complexity blocking functional launch

Phases that may benefit from spot research during planning:
